# Karpathy LLM 知识库

使用 LLM 构建和维护个人知识库。管理两个目录：原始材料目录 `<RAW_DIR>`（不可变）和知识库目录 `<WIKI_DIR>`（LLM 编译后的文章）。

核心理念（引自 Andrej Karpathy）：
- "LLM 负责编写和维护知识库；人类负责阅读和提问。"
- "知识库是一个持久的、不断累积的工件。"

## 架构

**`<RAW_DIR>`** — 不可变的原始材料。只能读取，永不能修改。按主题子目录组织。

**`<WIKI_DIR>`** — LLM 编译后的知识文章。完全所有权。按主题子目录组织，仅支持一级子目录。包含两个特殊文件：
- `index.md` — 全局索引。每个文章一行，按主题分组，包含链接、摘要、更新日期
- `log.md` — 追加式操作日志

## 初始化

仅在首次 Ingest 时触发。检查 `<RAW_DIR>` 和 `<WIKI_DIR>` 是否存在，仅创建缺失的部分，永不覆盖现有文件。

如果查询或 Lint 找不到知识库结构，告诉用户："请先运行 ingest 来初始化知识库。" 不要自动创建。

## Ingest（摄取）

将源内容保存到 `<RAW_DIR>`，然后编译到 `<WIKI_DIR>`。

### 获取（保存到原始材料目录）

1. 使用可用的工具获取源内容。无法访问源时要求用户直接粘贴内容。
2. 选择主题目录。检查现有子目录；主题接近则重用，仅为真正不同的主题创建新子目录。
3. 保存为 `<RAW_DIR>/<主题>/YYYY-MM-DD-描述性名称.md`。
   - 从源标题生成 slug，kebab-case，最多 60 字符
   - 发布日期未知则省略日期前缀
   - 同名文件存在则追加数字后缀
   - 包含元数据头：源 URL、收集日期、发布日期
   - 保留原始文本，清理格式噪音，不重写观点

### 编译（输出到 wiki 目录）

确定新内容的归属：
- **与现有文章核心论点相同** → 合并。添加新源到 Sources/Raw，更新受影响部分。
- **新概念** → 在相关主题目录创建新文章，按概念命名而非原始文件名。
- **跨多个主题** → 放最相关目录，在其他目录添加交叉引用。

检查事实冲突：新源与现有内容矛盾时，用源归属注释冲突。

文章格式要点：
- Sources 字段：作者、组织或出版物名称+日期，分号分隔
- Raw 字段：到原始材料文件的完整路径，分号分隔
- 使用绝对路径引用原始材料

### 级联更新

处理完主要文章后检查连锁效应：
1. 扫描同主题目录中受新源影响的文章
2. 扫描 `index.md` 中其他主题的相关概念文章
3. 更新所有受实质影响的文章，刷新 Updated 日期

### Ingest 后处理

更新 `index.md`：为每个处理过的文章添加或更新条目。添加新主题时包含一行描述。Updated 日期反映知识内容的最后更改。

追加到 `log.md`：
```
## [YYYY-MM-DD] ingest | <主要文章标题>
- Updated: <级联更新的文章标题>
```

## 查询

搜索知识库并回答问题。

1. 读取 `<WIKI_DIR>/index.md` 定位相关文章
2. 读取文章并综合答案
3. 优先使用知识库内容而非训练知识
4. 用路径引用源
5. 在对话中输出答案。除非用户要求，否则不写文件。

### 归档

用户明确要求归档时：
1. 将答案作为新 wiki 页写入 `<WIKI_DIR>/<主题>/`
2. 总是创建新页面，不合并到现有文章（归档内容是综合答案）
3. 更新 `index.md`，摘要前加 `[已归档]`
4. 追加到 `log.md`

## Lint（质量检查）

### 确定性检查（自动修复）

- **索引一致性** — 比较 `index.md` 与实际 wiki 文件：文件缺失则标记 `[缺失]`，索引缺失则添加条目
- **内部链接** — 检查 wikilinks 目标存在性，单匹配自动修复，零或多匹配报告
- **Raw 引用** — Raw 字段路径必须指向现有原始材料文件
- **交叉引用** — 添加相关文章间明显缺失的交叉引用

### 启发式检查（仅报告）

- 跨文章事实矛盾
- 过时声明
- 缺失冲突注释
- 孤儿页
- 缺失跨主题引用
- 经常提及但缺少专用页面的概念

### Lint 后处理

追加到 `log.md`：`## [YYYY-MM-DD] lint | 发现<N>个问题，<M>个已自动修复`

## 理解融合流程

这是 wiki 构建最关键的步骤。原始文本不能直接放入 wiki——必须经过理解→融合→重组。

```
原始材料（任意格式）
    ↓ [格式转换：markitdown / defuddle]
  MD 原始文本
    ↓
  阅读 → 理解内容 → 融合多源知识 → 重组为结构化文章
    ↓
  保存为 wiki 主文章（含 topic/sources/created frontmatter）
```

### 融合标准

- 一个主题一个目录
- 每主题 1~N 篇融合文章，覆盖概述、细节、版本演进等
- **融合密度 ≥ 70% 原始材料**：深入内容，不列文件清单
- 交叉引用：文章间通过链接建立关联
- 原始材料溯源：sources 字段标注源文件路径

### 目录编号规范

目录使用数字前缀统一编号，一经分配不轻易变动。格式：`XX-项目名`。

---

## 约定

- 所有地方使用标准 Markdown
- wiki 仅支持一级主题子目录
- 日志条目和日期使用当天日期
- Updated 日期反映知识内容的最后更改
- Ingest 会更新 `index.md` 和 `log.md`
- Archive（来自 Query）也会更新这两个文件
- Lint 更新 `log.md`（仅在自动修复索引时才更新 `index.md`）
- 普通查询不写入任何文件
- 所有路径使用绝对路径

## 配置

首次使用前和用户确认：

| 配置项 | 说明 |
|--------|------|
| `RAW_DIR` | 原始材料根目录（只读，永不修改） |
| `WIKI_DIR` | 知识库输出根目录（LLM 完全所有权） |

敏感信息（Wiki 账号、API Key 等）从环境变量读取，禁止硬编码。
